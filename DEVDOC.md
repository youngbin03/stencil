# 개발 문서 — SVG 템플릿 기반 AI 프레젠테이션 생성 시스템

> 코드네임(임시): **Stencil**
> 문서 버전: **v3** · 대상 독자: 개발 에이전트(Claude Code) + 1인 개발자
> 성격: **개발 사양서.** 구현 코드는 포함하지 않는다. 아키텍처·워크플로우·주요 로직·데이터 계약을 정의한다.
>
> **v3 변경 요약(중요):** 템플릿을 다시 export해 **텍스트가 진짜 `<text>`/`<tspan>`**로 들어왔다 — `font-family`·`font-size`·`font-weight`·`fill`이 모두 **실측값**으로 보존된다(Inter, Open Sans, Bricolage Grotesque, Neuton). 따라서 v2의 핵심 부담이 사라진다: **타입스케일·폰트·색은 더 이상 "추정"이 아니라 실측**이고, 폰트 수동 지정·글리프 높이 추정·A vs B 렌더 고민이 불필요하다. 다만 **`data-*` 라벨은 여전히 없고 역할은 `id`(Figma 레이어명)에만** 있다. 그래서 **(1)** 정규화기 **M0**는 "id→역할 매핑 + 실측 속성 읽기"만 하는 **경량 리더**로 축소되고, **(2)** v1 렌더 전략은 **"원본 SVG를 베이스로 두고 슬롯별 텍스트만 인플레이스 치환"**(원본 폰트·색·위치 그대로 유지)으로 확정한다. v1의 2층 토큰·구성 IR·결정론 솔버·어댑터 경계 철학은 **그대로 유지**한다.
>
> 입력 현황(실측): 3테마 black 40 / colorful 32 / green 11 = 83장, 전부 16:9. 단 `blackdesign/Frame-26·30·33` 3장은 `<text>` 없음(아웃라인-only 표지/장식 → decoration 처리). 31장은 래스터 `<image>` 포함.

---

## 0. 이 문서의 사용법

이 문서는 Claude Code가 프로젝트를 스캐폴딩하고 단계적으로 구현하기 위한 단일 진실 원천(SSOT)이다. 읽는 순서:

1. **1~3장**으로 무엇을/왜 만드는지와 핵심 개념을 잡는다.
2. **7장(데이터 계약)**을 가장 먼저 고정한다 — 모든 모듈이 이 계약 위에서 동작한다. 계약이 흔들리면 전부 흔들린다.
3. **6·8장**으로 모듈별 책임과 로직을 구현한다.
4. **11장 로드맵**의 순서를 반드시 지킨다. 수평이 아니라 **수직 슬라이스**로 진행한다.

핵심 설계 철학 한 줄: **"모델은 구성(composition)을 결정하고, 솔버는 배치(layout)를 풀고, 렌더러는 원본 디자인 위에 편집 가능한 텍스트만 얹는다. LLM은 절대 좌표를 출력하지 않는다."**

## 1. 프로젝트 개요

### 1.1 한 문장 정의
Figma에서 export된 프레젠테이션 SVG 템플릿을 **정규화**해 재사용 가능한 중간 표현(IR)으로 정립하고, 사용자 프롬프트를 받아 **그 디자인 일관성을 유지한 채** 새 슬라이드 덱을 생성하여, **원본 템플릿의 시각 디자인을 픽셀 그대로 보존하면서** 편집 가능한 텍스트를 얹어 웹에 렌더링하는 시스템.

### 1.2 핵심 비유 — 컴파일러
- **입력 SVG 템플릿(Figma `<text>` 보존)** = 소스 + 문법 정의
- **정규화(M0)** = 렉싱/전처리 → 슬롯 매니페스트(실측 속성) + 베이스 템플릿
- **추출(M1)** = 파싱 → 디자인 시스템 IR(중간 표현)
- **구성 엔진(Claude)** = 의미 해석 → 구성 IR
- **레이아웃 솔버** = 코드 생성(좌표/피팅 해결)
- **SVG 렌더러(인플레이스 치환 어댑터)** = 타깃 코드 출력

이 구조의 핵심 이점: IR을 자산으로 두면 **렌더러 어댑터를 교체**하는 것으로 출력 타깃을 바꿀 수 있다. v1 렌더 어댑터는 **인플레이스 텍스트 치환(A안)**이고, v2에서 **완전 재합성/테마 스왑(B안)** 어댑터를 형제로 추가한다. (텍스트가 실측 `<text>`라 B안도 v2에서 충분히 실현 가능.)

### 1.3 입출력 (확정)
- **입력:** Figma에서 export한 프레젠테이션 SVG 템플릿(슬라이드 1종 = 1파일). 텍스트가 `<text>`/`<tspan>`로 보존되고 폰트·크기·색이 실측되며 Figma 레이어명이 `id`로 보존된 상태를 전제한다. + 사용자 프롬프트 + 사용자 업로드 이미지.
- **출력:** 웹에 렌더링되는 슬라이드 덱(원본 SVG 베이스 + 슬롯별 텍스트 치환) + SVG 다운로드.

### 1.4 보유 입력 현황 (실측)
- `templates/` 아래 3개 테마: `blackdesign`(40), `colorfulldesign`(32), `greendesign`(11). 총 **83장**. 전부 `viewBox 0 0 1920 1080`(16:9).
- 텍스트가 진짜 `<text>`/`<tspan>`. `font-family`·`font-size`·`font-weight`·`fill`·`letter-spacing` 실측 보존. 폰트: Inter / Open Sans / Bricolage Grotesque / Neuton (역할·테마별).
- 역할은 `data-*`가 아니라 `id`(Figma 레이어명)에만: 텍스트(`Title, H1~H3, Headline, Subtitle, Body, Body Medium, Body 3, Text block, Caption, Label, Quote, Content, Number, List item, Metric, Date, PROJECT NAME, Team member`), 장식(`Decorative, Frame n, Rectangle n, Line`), 이미지(`Image, Profile pic`, 기기목업 iPhone/Macbook/Android), 차트(`Diagram, Pie slice, V Axis`).
- 83장 중 **31장**은 base64 래스터(`<image>`) 포함. `blackdesign/Frame-26·30·33` 3장은 `<text>` 없음(아웃라인-only → decoration 처리).
- **결정적 사실:** 텍스트가 실측 `<text>`이고 역할이 `id`에 있으니, M0는 "id→역할 매핑 + 속성 읽기"만 하면 된다. 토큰(폰트·크기·색)은 추정이 아니라 직접 읽힌다.

## 2. 목표 / 비목표

### 2.1 목표 (v1)
- Figma `<text>` 보존 SVG 템플릿 1종 이상을 **정규화**해 슬롯 매니페스트(실측 속성)를 산출하고, 이를 **디자인 시스템 IR**로 정립한다.
- 사용자 프롬프트(주제·맥락·분량) + 업로드 이미지를 받아, 템플릿의 블록/슬롯 어휘 안에서 **구성 IR**을 생성한다.
- 구성 IR을 **결정론적으로** 좌표/피팅으로 풀고, **원본 SVG를 베이스로 슬롯별 텍스트를 인플레이스 치환**해 슬라이드 덱 SVG를 렌더한다.
- 결과 덱을 웹에서 보고 SVG로 다운로드한다.
- 디자인 일관성(폰트·크기·색·간격·전체 비주얼)이 **원본 템플릿과 픽셀 단위로 동일**하게 유지된다.

### 2.2 비목표 (의도적 제외)
아래는 "아직 안 함"이지 "못 함"이 아니다. IR/어댑터 구조가 이들을 나중에 흡수하도록 설계하되, v1에서는 구현하지 않는다.

- **완전 재합성 렌더 / 테마 색 일괄 스왑(B안).** 토큰·블록으로 슬라이드를 처음부터 재구성하는 것은 v2 어댑터. v1은 원본 SVG를 베이스로 두고 텍스트만 치환한다(가장 높은 시각 충실도).
- **비텍스트 요소(도형·장식·차트)의 개별 편집.** v1 편집 대상은 텍스트 슬롯 + 이미지 슬롯뿐. 나머지는 원본 그대로 보존.
- **래스터(PNG·JPG) 템플릿 입력.** 입력은 벡터 SVG로 한정.
- **PPTX·DOCX·PDF 등 비-SVG 출력.** 렌더 어댑터로 후속 추가.
- **인브라우저 편집기.** v1은 뷰 + 다운로드만.
- **이미지 자동 생성.** v1은 사용자 업로드로 대체.
- **멀티 매체(포스터·사내 공문 등).**
- **네이티브 차트/표 생성 엔진.** v1은 차트=이미지 업로드, 표=단순 텍스트 그리드.
- **실시간 협업·버전 관리·팀 권한·결제.**

## 3. 핵심 개념 (용어집)

| 용어 | 정의 |
|---|---|
| **정규화기(M0)** | Figma `<text>` 보존 SVG를 받아 `id`→역할을 닫힌 어휘로 매핑하고 각 슬롯의 실측 속성을 읽어 **슬롯 매니페스트**를 만드는 경량 리더. 원본 SVG는 렌더 베이스로 함께 보존. M1/M2의 입력을 만든다. |
| **베이스 템플릿** | 원본 SVG 그 자체(편집 가능 `<text>` 포함). 렌더 시 베이스로 깔고 슬롯 텍스트만 치환한다. 별도 가공·텍스트 제거 불필요. |
| **슬롯 매니페스트** | 각 슬롯의 `{id, role, type, bbox, color, fontFamily, fontSize, fontWeight, letterSpacing, align}` 목록(텍스트는 전부 실측, 이미지는 bbox/ratio). 솔버·렌더러의 치환 근거. |
| **디자인 시스템 IR** | 정규화 산출물에서 정립한 영속 자산. 토큰(1층) + 블록 + 레이아웃 + 베이스 템플릿 참조로 구성. 템플릿당 1회 생성. |
| **구성 IR** | 생성 요청마다 Claude가 출력하는 산출물. "어느 레이아웃에 어떤 블록을 무슨 콘텐츠로". **좌표 없음.** |
| **토큰(1층)** | 매체 무관 디자인 언어 — 팔레트, 타입스케일(실측), 폰트, 간격. **"일관성"의 정체.** |
| **블록(2층)** | 매체 의존 컴포넌트 패턴 — 제목 블록, 스탯 카드, 이미지+캡션 등. 내부 슬롯을 가짐. |
| **슬롯(slot)** | 블록 안에서 콘텐츠가 채워지는 칸. 역할(role)과 제약(max, ratio 등), 기하(bbox)를 가짐. |
| **레이아웃(layout)** | 한 종류의 슬라이드 형판. 영역(region) + 각 영역 허용 블록을 정의. SVG 템플릿 1파일 = 1레이아웃. |
| **영역(region)** | 레이아웃 내 배치 단위. 위치·흐름방향(row/column)·간격을 가짐. |
| **닫힌 어휘** | 정규화기가 Figma `id`를 매핑하는 표준 역할 집합(7.1). |
| **레이아웃 솔버** | 구성 IR + 영역/슬롯 기하 → 실제 좌표를 계산하는 결정론적 모듈. **안정성의 원천.** |
| **인플레이스 치환 렌더러(A안)** | 원본 SVG를 베이스로 두고 슬롯별 `<text>` 내용·피팅만 교체해 슬라이드 SVG를 만드는 v1 렌더 어댑터. 폰트·색·위치는 원본 그대로. |

**두 층 분리 원칙:** 일관성은 1층(토큰)에서 오고 적응은 2층(블록·레이아웃)에서 온다. 정규화·IR·렌더러 전부에서 이 둘을 섞지 않는다.

## 4. 전체 아키텍처

### 4.1 파이프라인 (단방향)

```
[Figma SVG]  (텍스트=<text>, 폰트·색 실측, id=레이어명)
        │
        ▼
  ┌───────────────┐
  │ M0. 정규화기   │  id→닫힌어휘 매핑 · 슬롯 실측속성 읽기
  └───────────────┘   → 슬롯 매니페스트 (+ 베이스 템플릿 보존)
        │
        ▼
  ┌───────────────┐
  │ M1. 추출기     │  매니페스트 → 토큰 + 블록 + 레이아웃
  │ M2. 계약 검증  │  (정규화 산출물이 7.1 충족하는지 게이트)
  └───────────────┘
        │
        ▼
  [디자인 시스템 IR] + [베이스 템플릿] ── (Supabase 영속: 템플릿당 1회)
        │
        │   [사용자 프롬프트 + 업로드 이미지]
        │            │
        ▼            ▼
  ┌─────────────────────┐
  │ M3. 구성 엔진(Claude) │  블록 어휘 안에서 구성 결정 (tool use)
  └─────────────────────┘
        │
        ▼
   [구성 IR]  ← 좌표 없음
        │
        ▼
  ┌───────────────┐
  │ M4. 레이아웃 솔버│  슬롯 기하 + 흐름 → 좌표(px) + 텍스트 피팅
  └───────────────┘
        │
        ▼
  ┌──────────────────────────────┐
  │ M5. 렌더러(인플레이스 치환)    │  원본 SVG 베이스 + 슬롯 텍스트 치환
  └──────────────────────────────┘
        │
        ▼
  ┌───────────────┐
  │ M6. 웹앱/API    │  뷰 · 다운로드 (v2: 편집)
  └───────────────┘
```

### 4.2 불변 원칙 (위반 금지)
1. **단방향.** 각 단계는 앞 단계 산출물만 소비한다. 역방향 의존 금지.
2. **LLM은 좌표를 만들지 않는다.** Claude 출력은 항상 "구성"이고, 픽셀은 솔버가 푼다.
3. **계약 우선.** 모듈 간 인터페이스는 7장 스키마로만. 한 모듈을 통째 교체해도 계약이 같으면 나머지는 안 바뀐다.
4. **결정론 우선.** 같은 IR + 같은 구성 IR은 항상 같은 SVG. 무작위성은 M3(Claude)에만 격리.
5. **원본 보존.** v1 렌더러는 베이스 템플릿의 비텍스트 요소를 변형하지 않는다. 슬롯 텍스트만 치환한다.

## 5. 워크플로우 — 주요 단계

| # | 단계 | 비고 |
|---|---|---|
| 0 | **정규화(M0)** | Figma SVG → 슬롯 매니페스트(실측) + id매핑 + 베이스 템플릿 보존. 사람이 매핑 1회 확인·보정 가능 |
| 1 | **계약 검증(M2)** | 정규화 산출물이 7.1을 충족하는지 게이트. 위반 시 리포트 |
| 2 | **추출(M1) → 디자인 시스템 IR** | 토큰(색·폰트·크기·간격 전부 실측), 블록·레이아웃 다음 |
| 3 | **프롬프트 + 이미지 입력** | 이미지는 슬롯에 바인딩 |
| — | **구성 엔진(Claude) → 구성 IR** | tool use로 스키마 강제 |
| — | **레이아웃 솔버 → 좌표/피팅 해결** | 결정론 |
| 4 | **렌더(인플레이스 치환) → 웹 뷰 + 다운로드** | 원본 베이스 + 텍스트 치환. 덱 단위 |
| 5 | **v1: 다운로드 / v2: 인브라우저 편집** | 편집은 비목표(v1) |

## 6. 모듈별 사양

각 모듈은 `책임 / 입력 / 출력 / 주요 로직 / 엣지케이스`로 기술한다. 구체 알고리즘은 8장 참조.

### M0. 정규화기 (Figma SVG → 슬롯 매니페스트 + 베이스 템플릿) — 경량
- **책임:** `data-*` 라벨이 없는 Figma export SVG를 후속 모듈이 소비 가능한 형태로 전처리. **텍스트가 실측 `<text>`이므로 무거운 재구성 없이 읽기/매핑만 한다.**
- **입력:** Figma SVG(텍스트=`<text>`/`<tspan>`, id=레이어명).
- **출력:** ① 슬롯 매니페스트(7.4) ② id→역할 매핑 로그 ③ 베이스 템플릿(원본 SVG 그대로 — 렌더 베이스).
- **주요 로직:**
  - SVG를 XML/DOM으로 파싱. `viewBox` 기준 px 정규화.
  - **슬롯 식별 + 실측 읽기:** `<text>` 노드(및 `<image>`/기기목업)를 슬롯 후보로 수집. 각 텍스트 슬롯에서 `font-family`·`font-size`·`font-weight`·`fill`·`letter-spacing`·`x`/`y`를 **직접 읽고**, 폭/높이는 헤드리스 `getBBox` 또는 텍스트 메트릭으로 측정해 `bbox` 산출. 정렬은 `text-anchor` 또는 x 분포로 판정.
  - **역할 매핑:** Figma `id`(`H2`, `Body Medium_3`, `PROJECT NAME` 등)를 닫힌 어휘로 매핑(사전 + 접미사 `_n`·공백·대소문자 정규화). 미매칭은 `decoration` 폴백 + 경고.
  - **베이스 템플릿 보존:** 원본 SVG는 가공하지 않고 그대로 보관(렌더 시 베이스). 슬롯의 `id`로 렌더러가 해당 `<text>`를 찾아 내용·피팅을 치환한다. 텍스트 제거/배경 분리 불필요.
- **엣지케이스:** 모호한 슬롯 → 매니페스트에 `uncertain` 플래그(사람 확인). `<text>`가 0개인 파일(`Frame-26·30·33`) → 전체 decoration(텍스트 슬롯 없음)으로 처리. `<image>` 래스터 → 사용자 교체 대상이면 `image` 슬롯 등록(bbox+ratio), 아니면 decoration.

### M1. 추출기 (정규화 산출물 → 디자인 시스템 IR)
- **책임:** 슬롯 매니페스트에서 토큰·블록·레이아웃 산출.
- **입력:** M0 산출물(여러 레이아웃 SVG 다수 가능).
- **출력:** 디자인 시스템 IR(7.2).
- **주요 로직:**
  - **토큰 추출(전부 실측):** 슬롯·배경의 fill/stroke 색을 빈도·면적 가중 집계 → 팔레트(bg/text/accent). 슬롯의 실측 `fontSize`·`fontFamily`·`fontWeight` 분포를 역할별로 묶어 타입스케일. 슬롯 간격 분포 → spacing unit. (추정 없음. `estimated`는 기본 false.)
  - **블록 추출:** 근접·정렬·역할 패턴으로 슬롯을 블록 인스턴스로 군집(8.2). bbox·슬롯 제약 기록. (v1은 명시 `data-block` 없으므로 휴리스틱 + 수동 보정 허용.)
  - **레이아웃 추출:** 영역(region) 경계를 슬롯 분포로 추정하거나 수동 정의. 영역별 허용 블록 구성.
- **엣지케이스:** 한 역할의 폰트가 파일마다 다르면(예: Inter vs Open Sans) 슬롯별 실측값을 우선하고 토큰은 대표값으로 둔다. getBBox 측정 실패 → 휴리스틱 폭 폴백 + 경고.

### M2. 라벨 계약 검증기
- **책임:** M0 산출물(슬롯 매니페스트)이 7.1 계약을 지키는지 게이트.
- **출력:** 통과 / 위반 리포트.
- **주요 로직:** 매니페스트 필수 필드(role·bbox) 존재, 역할이 닫힌 어휘인지, 영역-블록 정합성, `uncertain` 슬롯 잔존 여부 검사.

### M3. 구성 엔진 (Claude API)
- **책임:** 사용자 의도를 템플릿 어휘로 번역해 구성 IR 생성.
- **입력:** 디자인 시스템 IR의 **어휘 요약**(레이아웃 목록 + 블록·슬롯·제약; 토큰 상세 불필요) + 프롬프트 + 업로드 이미지 메타.
- **출력:** 구성 IR(7.3).
- **주요 로직 (2패스):**
  - **패스 A — 덱 아웃라인:** 프롬프트 → 슬라이드 수·각 슬라이드 목적·레이아웃 선택.
  - **패스 B — 슬라이드 채움:** 슬라이드별 블록 선택·시퀀싱·슬롯 콘텐츠. 슬롯 `max`를 지시에 명시.
  - **출력 강제:** Claude **tool use**로 구성 IR JSON 스키마 강제(산문 금지).
- **불변:** 좌표·색값·폰트크기 직접 지정 금지. role/블록/콘텐츠만.
- **엣지케이스:** 허용 안 된 블록 선택 → 검증 리젝트 후 1회 재시도. max 초과 → 솔버 피팅에 위임(8.4).

### M4. 레이아웃 솔버 (결정론적)
- **책임:** 구성 IR + 슬롯/영역 기하 → 절대 좌표 + 텍스트 피팅.
- **입력:** 구성 IR + 슬롯 매니페스트 기하(+ 영역 정의).
- **출력:** 렌더 가능 트리(각 요소의 x/y/w/h/스타일 확정).
- **주요 로직:**
  - **고정 슬롯 모드(v1 기본):** 각 슬롯은 매니페스트 bbox를 그대로 점유. 콘텐츠를 그 칸에 피팅. (원본 레이아웃 보존이 최우선이므로 v1은 흐름 재배치보다 슬롯 직접 매핑을 우선.)
  - **흐름 배치(반복 블록용):** `flow=column`은 위→아래 gap 스택, `flow=row`는 좌→우 분배. 반복 블록은 콘텐츠 수만큼 복제.
  - 텍스트 피팅(8.4): 줄바꿈·오토핏 축소·말줄임.
- **불변:** 무작위성 없음. 동일 입력 → 동일 출력.

### M5. SVG 렌더러 (인플레이스 치환 어댑터) — 변경
- **책임:** 렌더 트리 + 베이스 템플릿 → 슬라이드 SVG(슬라이드당 1개). **원본 SVG를 베이스로 두고 슬롯 텍스트만 치환.**
- **주요 로직:**
  - 베이스 템플릿 SVG를 파싱해, 슬롯 `id`로 해당 `<text>`를 찾아 내용(`<tspan>` 줄들)을 솔버 피팅 결과로 교체. `font-family`·`font-size`·`fill`·`x`/`y`·`text-anchor`는 원본 유지(피팅이 size를 줄였으면 그 값으로 갱신). **path 아웃라인 금지.**
  - 비텍스트 요소(도형·장식·차트·기기목업)는 손대지 않는다.
  - 각 치환 `<text>`에 `data-role` 보강(편집 UI·디버깅용).
  - `image` 슬롯 교체 시 원본 래스터/플레이스홀더를 사용자 `<image>` + `clipPath`로 교체.
  - (선택) 색을 CSS 변수/클래스로 노출해 v2 테마 스왑 여지를 남김.
- **출력 단위:** 슬라이드별 SVG. 덱은 SVG 배열.
- **어댑터 경계:** v2 완전 재합성/테마 스왑(B안) 렌더러를 형제로 추가할 수 있도록 `RenderAdapter` 인터페이스를 유지.

### M6. 웹앱 / API
- **책임:** 템플릿 업로드·정규화/추출 결과 확인·프롬프트 입력·생성 트리거·덱 뷰·다운로드.
- **v1 화면:** ① 템플릿 관리(업로드→정규화→매핑 확인/보정→추출→IR 확인), ② 생성(프롬프트+이미지→덱), ③ 덱 뷰어(SVG 렌더 + 다운로드).
- **v2:** 인브라우저 SVG 편집 + 완전 재합성 렌더.

## 7. 데이터 계약 (스키마)

> 이 장의 스키마는 모든 모듈의 인터페이스다. **여기를 먼저 고정**하고 구현을 시작한다. 예시는 형태 설명용이며 필드는 확장 가능.

### 7.1 라벨 계약 — 두 단계로 분리

v1 입력은 라벨이 없으므로, 계약을 **(A) 원본 Figma 입력 규약**과 **(B) 정규화 산출 규약**으로 나눈다. M2는 (B)를 검증한다.

**(A) 원본 Figma 입력 규약 (느슨 — 권고)**
- SVG 1파일 = 1레이아웃(슬라이드 1종). 루트 `<svg>`에 `viewBox` 필수.
- 텍스트는 **`<text>`/`<tspan>`로 유지**(아웃라인 path 아님). `font-family`·`font-size`·`font-weight`·`fill` 보존.
- 텍스트 레이어는 Figma에서 **의미 있는 레이어명**을 갖는다(`Title`, `Body`, `Caption`, `H2` 등). export 시 이 이름이 `id`로 보존된다.
- `data-*` 라벨은 없어도 된다(현 입력 전제). 역할은 `id`로 추론한다.

**(B) 정규화 산출 규약 (엄격 — M2가 검증)**
- 모든 텍스트 슬롯은 닫힌 어휘 `role`을 갖는다: `title, subtitle, eyebrow, headline, body, bullet, caption, quote, label, kpi, image, table, logo, footer, pagenum, divider, decoration`.
- 각 슬롯은 `bbox`(viewBox px) 필수. 텍스트 슬롯은 `color`·`fontSize`·`fontFamily`·`align` 보유(전부 실측).
- `decoration`/`divider`는 생성에서 무시(원본 보존만).
- `uncertain` 플래그가 남은 슬롯이 없어야 통과(사람 보정 필요).
- 제약: `max`(텍스트 글자수), `ratio`(이미지 16:9 등)는 선택.

> 원본 `id`→`role` 매핑 사전과 정규화 로직은 8.0.

### 7.2 디자인 시스템 IR (영속 자산)

```json
{
  "template_id": "string",
  "theme": "colorful | black | green",
  "version": 2,
  "canvas": { "w": 1920, "h": 1080 },
  "base_template": { "layout_id": "colorful_01", "url": "storage://templates/colorful_01.svg" },
  "tokens": {
    "fontFamily": "Inter",
    "colors": { "primary": "#1A1A2E", "accent": "#E94560", "bg": "#FFFFFF", "text": "#000000" },
    "type": {
      "title":    { "family": "Bricolage Grotesque", "size": 180, "weight": 200, "lineHeight": 1.0, "estimated": false },
      "subtitle": { "family": "Inter", "size": 40, "weight": 600, "lineHeight": 1.2, "estimated": false },
      "body":     { "family": "Inter", "size": 28, "weight": 400, "lineHeight": 1.4, "estimated": false }
    },
    "spacing": { "unit": 8, "scale": [8, 16, 24, 48, 96] }
  },
  "blocks": [
    {
      "id": "stat_card",
      "bbox": { "x": 0, "y": 0, "w": 360, "h": 200 },
      "repeatable": true,
      "slots": [
        { "role": "kpi",     "type": "text",  "max": 8 },
        { "role": "caption", "type": "text",  "max": 40 }
      ]
    }
  ],
  "layouts": [
    {
      "id": "colorful_01",
      "regions": [
        { "id": "header", "bbox": { "x": 96, "y": 80,  "w": 1728, "h": 200 }, "flow": "column", "gap": 16, "allowed_blocks": ["title", "subtitle"] },
        { "id": "row",    "bbox": { "x": 96, "y": 360, "w": 1728, "h": 480 }, "flow": "row",    "gap": 48, "allowed_blocks": ["stat_card"] }
      ]
    }
  ]
}
```

- `base_template`로 레이아웃별 원본 SVG를 참조한다. 렌더러가 이를 베이스로 두고 슬롯 텍스트만 치환한다.
- `type.*`의 `family`/`size`/`weight`는 `<text>`에서 직접 읽은 **실측값**(`estimated:false`). 역할별 폰트가 다를 수 있어 `family`를 토큰에 포함.
- `tokens.fontFamily`는 본문 기본 폰트(대표값). 슬롯별 실측 폰트가 우선.

## 8. 주요 로직 상세

### 8.0 정규화 — id→역할 매핑 & 실측 읽기 (M0)
1. **정규화 키:** `id`에서 접미사 `_\d+` 제거, 공백→단일, 소문자화 → 매핑 키. 예: `Body Medium_3` → `body medium`, `PROJECT NAME` → `project name`.
2. **매핑 사전(초안, 보정 가능):**
   - `presentation title`, `title`, `h1` → `title`
   - `headline`, `h2` → `headline` (또는 `subtitle` — 테마별 보정)
   - `subtitle`, `h3 regular`, `h3 medium`, `subtitle+body` → `subtitle`
   - `project name`, `date` → `eyebrow`
   - `body`, `body medium`, `body 3`, `text block`, `textblock`, `text`, `content`, `list item`, `team member`, `header` → `body`
   - `caption`, `label`, `number`, `metric`, `annotate` → `caption`
   - `quote` → `quote`
   - `line`, `v axis` → `divider`
   - `decorative`, `frame n`, `rectangle n`, `group`, `vector`, `shape`, `diagram`, `pie slice` → `decoration`
   - `image`, `profile pic`, `screen **insert designs here**`, 기기목업(`iphone…`, `macbook…`, `android…`) → `image` (교체 대상이면 슬롯, 아니면 decoration)
   - 미매칭 → `decoration` 폴백 + `uncertain:true` 경고.
3. **실측 읽기:** 각 `<text>` 슬롯에서 `font-family`·`font-size`·`font-weight`·`fill`·`letter-spacing`·`x`/`y`를 직접 읽는다. 추정 없음.
4. **bbox 측정:** 헤드리스 `getBBox`(예: `resvg-js`/브라우저) 또는 폰트 메트릭으로 `<text>`의 폭/높이 산출. x/y(+ `text-anchor`)로 좌상단 정규화.
5. **정렬:** `text-anchor`(`start`/`middle`/`end`) → `left`/`center`/`right`. 없으면 x 분포로 보조 판정.
6. **베이스 템플릿 보존:** 원본 SVG는 가공하지 않고 그대로 저장. 슬롯 `id`가 렌더 시 치환 앵커.
> M0 품질은 사전 정확도에 의존한다. v1은 **매핑 결과를 사람이 1회 확인·보정**하는 단계를 둔다(M6 템플릿 관리 화면).

### 8.1 추출 — 토큰 (1층, 전부 실측)
1. 슬롯·배경 fill/stroke 색을 빈도·면적 가중으로 상위 N → 팔레트. 최대 배경 면적 = `bg`, 본문 슬롯색 = `text`, 강조 빈도색 = `accent`.
2. 슬롯 실측 `fontSize`·`fontFamily`·`fontWeight`를 역할별로 묶어 title/subtitle/body 타입스케일(`estimated:false`).
3. 슬롯 bbox 간격 최빈값 → spacing unit, 배수 분포 → scale.
> 색·폰트·크기·간격 전부 실측이라 신뢰 높음. 추출 구현 첫 타깃.

### 8.2 추출 — 블록·레이아웃 (2층)
1. v1 입력은 명시 `data-block`이 없으므로 **근접·정렬·역할 반복 패턴**으로 슬롯을 블록 후보로 군집(예: `kpi`+`caption`이 규칙적 간격으로 반복 → `stat_card` repeatable).
2. 군집 결과는 **수동 보정 허용**(템플릿 관리 화면). v1은 휴리스틱 + 사람 확인 조합.
3. 영역은 슬롯 분포로 경계 추정하거나 수동 정의, 포함 블록으로 `allowed_blocks` 구성.
> 블록 추출은 v1에서 가장 불확실. 그래서 "고정 슬롯 모드"(8.4 솔버)를 기본으로 두어 블록 추출 없이도 동작하게 한다.

### 8.3 구성 엔진 — 2패스 + tool use
- **어휘 요약 주입:** Claude에 토큰 상세 대신 `레이아웃 id 목록 + 영역·허용 블록 + 블록 슬롯·제약`만 전달.
- **패스 A(아웃라인):** "프롬프트를 N장으로, 각 장의 목적과 layout_id".
- **패스 B(채움):** 각 장 영역별 블록 배열 + 슬롯 콘텐츠. 슬롯 `max`를 시스템 지시에 포함.
- **스키마 강제:** 구성 IR을 tool 입력 스키마로 정의, tool use로 받음.
- **검증 루프:** (a) 허용 블록만 (b) 필수 슬롯 채움 검사 → 실패 시 오류와 함께 1회 재요청.

### 8.4 텍스트 피팅 / 오버플로 (솔버 내부)
순서대로 적용:
1. **줄바꿈:** 슬롯 폭 기준 단어 단위 래핑. 글자폭은 폰트 메트릭 또는 v1 휴리스틱(평균 글자폭 계수).
2. **오토핏 축소:** 래핑 후 높이가 슬롯 bbox 초과 시 폰트 size step 축소(`min-size`까지).
3. **말줄임:** 그래도 넘치면 말줄임(…) + **경고 플래그** 첨부.
> Claude `max`가 1차 방어, 솔버 피팅이 2차. 슬롯 bbox가 고정이므로 피팅이 특히 중요하다.

### 8.5 이미지(사용자 업로드) 처리
- 업로드 → Supabase Storage → `asset_id` 발급.
- `image` 슬롯 `ratio`에 맞춰 커버 크롭 좌표 계산(중앙 기준). 원본 보존.
- 렌더러는 베이스 템플릿의 기존 래스터/플레이스홀더를 사용자 `<image>` + `clipPath`로 교체. 누락 시 원본 유지 또는 플레이스홀더 + 경고.

## 9. 기술 스택 & 근거

> 출력이 SVG로 확정되어 python-pptx 의존이 없다. **전 구간 단일 언어(TypeScript)** — 1인 개발의 가장 큰 마찰(언어 점프)을 제거한다.

| 레이어 | 선택 | 근거 |
|---|---|---|
| 코어 파이프라인(M0·M1·M3·M4·M5) | **TypeScript (Next.js Route Handlers / Node)** | SVG는 XML, JS DOM 파서로 충분. Anthropic TS SDK. 단일 언어. |
| SVG 파싱/직렬화 | DOM/XML 파서 (예: `@xmldom`/브라우저) | 결정론적 읽기·치환. 텍스트가 `<text>`라 path 파싱 불필요. |
| bbox/텍스트 측정 | 헤드리스 `getBBox`(예: `resvg-js`/브라우저) 또는 `opentype.js` 메트릭 | M0 bbox + 솔버 피팅. |
| 프론트엔드 | **Next.js 15 + TypeScript** | 주력 스택. |
| DB·스토리지·인증 | **Supabase** | IR=JSONB, 베이스 템플릿 SVG·이미지=Storage. |
| LLM | **Anthropic API (tool use)** | 구성 IR 스키마 강제. |
| 비동기 | v1 **동기 처리**. 느려지면 큐 추가. | 과설계 방지. |

> 폰트: 원본 `<text>`의 `font-family`가 실측되므로 그대로 사용한다(Inter, Open Sans, Bricolage Grotesque, Neuton). 이들을 웹폰트(`@font-face`/구글폰트)로 임베드. 한글 콘텐츠 대응 폴백 폰트(예: Pretendard)는 별도 지정.

## 10. 리포지토리 구조 (제안)

```
stencil/
├─ apps/
│  └─ web/                  # Next.js 15 (UI + API Route Handlers)
│     ├─ app/
│     │  ├─ templates/      # 업로드·정규화·매핑 보정·추출·IR 확인
│     │  ├─ generate/       # 프롬프트·이미지·생성
│     │  └─ decks/          # 덱 뷰어·다운로드
│     └─ api/               # route handlers (normalize, extract, compose, render)
├─ packages/
│  ├─ ir/                   # 데이터 계약(타입) — 7장 스키마의 단일 정의처
│  ├─ normalizer/           # M0: Figma SVG → 슬롯 매니페스트 + 베이스 템플릿
│  ├─ extractor/            # M1·M2: 정규화 산출물 → 디자인 시스템 IR + 검증
│  ├─ composer/             # M3: Claude tool use → 구성 IR
│  ├─ solver/               # M4: 구성 IR → 좌표 + 텍스트 피팅
│  └─ renderer/             # M5: 렌더 트리 → SVG (어댑터: inplace[v1] / resynth[v2])
├─ fixtures/
│  └─ templates/            # 샘플 Figma SVG (= 현 templates/, 개발·테스트용)
└─ docs/
   └─ DEVDOC.md             # 이 문서
```

- `packages/ir`이 7장 계약의 **단일 타입 정의처**. 모든 패키지가 의존.
- `renderer`는 어댑터 패턴: v1 `inplace`(인플레이스 치환), v2 `resynth`/`pptx`를 형제로 추가.
- 현재 `templates/` 3테마 SVG가 `fixtures/templates`의 시드.

## 11. 개발 로드맵 — 수직 슬라이스

> **원칙:** 수평(모듈 완성도순)이 아니라 수직(끝에서 끝까지 가늘게). 각 Phase 끝에 "입력→출력 한 바퀴"가 돈다. 가장 불확실한 것부터.

**Phase 0 — 정규화 PoC (가장 먼저, 가장 불확실).**
- `normalizer`로 `colorfulldesign/Frame-0.svg` 1장을 슬롯 매니페스트(role·bbox·color·fontFamily·fontSize 실측)로 읽고, 베이스 템플릿(원본 SVG)을 보존.
- ✅ 검증: 슬롯 id→역할 매핑이 맞고, bbox가 원래 `<text>` 자리와 맞는가.

**Phase 1 — 계약 + 골격.**
- `packages/ir`에 7장 스키마(7.2~7.4)를 타입으로 고정.
- 손으로 만든 구성 IR + Phase 0 산출물로 **M4 솔버 → M5 인플레이스 치환 렌더러**를 연결해 SVG 1장 출력.
- ✅ 검증: 원본 SVG 베이스에서 슬롯 텍스트가 새 내용으로 치환되고 폰트·색·위치가 그대로 유지되는가.

**Phase 2 — 정규화 자동화 + 추출(M0·M1·M2).**
- id→역할 매핑 사전 적용, 색·폰트·간격 토큰 추출(전부 실측), 83장 일괄 정규화.
- 계약 검증기 + 매핑 보정 흐름.
- ✅ 검증: 손IR 없이 SVG에서 디자인 시스템 IR이 나온다.

**Phase 3 — 구성 엔진(M3).**
- Claude tool use로 프롬프트 → 구성 IR. 단일 레이아웃부터. 검증 루프.
- ✅ 검증: 프롬프트 → 구성 IR → (솔버·인플레이스 렌더) → SVG.

**Phase 4 — 풀 파이프 + 멀티 슬라이드.**
- 2패스, 멀티 블록·반복 블록, 텍스트 피팅, 사용자 이미지 바인딩.
- ✅ 검증: "주제 한 줄 → 여러 장 덱".

**Phase 5 — 웹앱 셸(M6).**
- Next.js + Supabase: 템플릿 업로드 → 정규화/매핑 보정 → 추출 → 생성 → 뷰 → 다운로드 한 바퀴.

**Phase 6 — 다듬기.**
- 3테마 전체, 오버플로 엣지케이스, 폰트 임베드, 경고 UX.

**(v2 이후)** 인브라우저 편집기 · 완전 재합성 렌더 어댑터(B안) · PPTX 어댑터 · 멀티 매체 · 이미지 생성.

## 12. 리스크 & 미해결 결정

**핵심 리스크**
1. **역할 매핑 정확도(M0)** — `data-*`가 없어 `id`→역할 매핑이 틀리면 콘텐츠가 엉뚱한 슬롯에 들어간다. → 사전 + **사람 1회 보정** 게이트. `uncertain` 플래그.
2. **텍스트 피팅 정확도** — 슬롯 bbox 고정이라 새 텍스트가 길면 오버플로. → `max` + 솔버 피팅(오토핏 축소·말줄임), 필요 시 정밀 측정으로 격상.
3. **폰트 임베드** — 실측 폰트(Bricolage Grotesque 등)가 뷰어에 없으면 대체로 깨짐. → 웹폰트 임베드 + 한글 폴백.
4. **bbox 측정 정확도** — `<text>`는 w/h가 없어 getBBox/메트릭 측정에 의존. → 헤드리스 측정, 실패 시 휴리스틱 폴백 + 경고.
5. **텍스트 0개 파일** — `Frame-26·30·33` 등 표지/장식은 슬롯이 없다. → decoration-only 레이아웃으로 처리(생성 시 텍스트 슬롯 없음).

**미해결 결정(추후 확정)**
- id→역할 매핑 사전의 테마별 최종 확정(특히 `headline`/`subtitle` 경계, `number`/`metric`의 caption vs kpi).
- 블록 자동 군집을 어디까지 신뢰할지 vs 고정 슬롯 모드만으로 갈지.
- 덱 다운로드 형식: 슬라이드별 SVG 다발 vs zip vs 후속 PPTX.
- v2 완전 재합성/테마 스왑 렌더(B안) 착수 시점.

## 13. Claude Code 작업 지침

1. **계약 먼저.** 7장 스키마(7.2~7.4)를 `packages/ir`에 타입으로 구현하기 전에는 다른 모듈을 만들지 않는다.
2. **수직 슬라이스.** 11장 Phase 순서를 지킨다. Phase 0(정규화 PoC)가 최우선.
3. **LLM은 좌표를 만들지 않는다.** 구성 엔진 출력에 좌표·색·폰트크기가 들어가면 설계 위반.
4. **결정론 격리.** 무작위성은 M3에만. M0·M4·M5는 같은 입력에 같은 출력.
5. **원본 보존.** v1 렌더러는 베이스 템플릿의 비텍스트 요소를 변형하지 않는다(슬롯 텍스트만 치환).
6. **어댑터 경계 보존.** 렌더러는 v2 재합성(B안)이 형제로 붙도록 인터페이스 유지.
7. 각 Phase 종료 시 `fixtures`(=`templates/`) 샘플로 한 바퀴 돌려 회귀 확인.
